# 建议实现的业务场景（详细版，含 MCP 封装清单）

**日期**：2026-07-10
**说明**：每个场景给出用户话术、Agent 执行链路、需封装的 MCP tool、右栏联动与验收标准；文末附按优先级汇总的 MCP 封装总清单。工具命名与《技术实现路线推荐》§二 保持一致。

**通用设计约定**（所有 tool 遵守）：

- 每个 tool = 一个原子业务能力，粒度标准：一句自然语言能说清它干什么，且参数可由上下文推断。
- 入参出参均带完整 JSON Schema；空间数据统一 GeoJSON（CGCS2000 / EPSG:4490）。
- 分钟级以上的长任务一律拆"提交 + 查询"两个 tool（`xxx_submit` 返回 task_id，`get_task_status` 轮询），不同步阻塞。
- 标注 🔒 的 tool 为高危写操作：Agent 只能生成待确认指令，人工点击确认后才真正执行（人在环）。
- tool 的 name/description/参数说明是 Agent 命中率的第一杠杆，按"给一个不了解系统的新同事看"的标准写。

## P0：7月22日深圳演示必做（四场景串成一条主线）

> **演示主线**："帮我查一下光明区的图斑" → 调度周边无人机 → 自动规划航线（多图斑覆盖）→ 嵌入界面人工微调 → Agent 解释航线逻辑 → "我要起飞" → 飞前检查（CoT 展示）→ 人工确认放飞。

### 场景 1：图斑查询与无人机调度

**用户话术示例**（评测集要覆盖这些变体）：
- "帮我查一下光明区最新下发的图斑"
- "这周有哪些图斑要核查？" / "查一下 GM-03 这个图斑的情况"
- "这些图斑周边有哪些无人机可以用？" / "调度一下附近的无人机"

**Agent 执行链路**：
1. 意图理解 + 指代消解（"这些图斑"指上文查询结果；"附近"默认半径 5km）
2. 调 `query_plots` → 图斑列表落右栏地图
3. 调 `find_nearby_drones` → 机标落图
4. 综合电量/距离/挂载给出调度建议（这一步是纯 LLM 推理，不是工具）
5. 用户认可后调 `dispatch_drone` 🔒 锁定无人机

**需封装的 MCP tool**（drone-dispatch-mcp）：

| tool | 入参 | 返回 | 对接系统 |
|---|---|---|---|
| `query_plots` | region?（行政区名）, plot_ids?[], plot_type?（疑似变化类型）, date_range?, batch_no?（下发批次） | 图斑数组：id、类型、面积、优先级、GeoJSON 边界、下发时间 | 图斑管理库 |
| `find_nearby_drones` | plot_id 或 location(GeoJSON point), radius_km（默认 5） | 无人机数组：id、机型、位置、电量、挂载、状态（空闲/任务中/维保）、距目标距离 | 设备管理 / 大疆上云 API 物模型 |
| `get_drone_status` | drone_id | 单机详情：电量、位置、固件、避障开关状态、健康自检 | 大疆上云 API |
| `dispatch_drone` 🔒 | drone_id, task_type, plot_ids[] | 调度单 id、锁定状态 | 任务调度模块 |

**右栏联动**：`show_map`（图斑多边形 → 追加机标+电量标签）。
**验收标准**：评测集 10 条话术，工具命中率 ≥ 90%，region/radius 传参正确率 ≥ 90%；地图落图 ≤ 2s。
**难点**：区域名/图斑编号的指代消解；"附近""周边"等模糊半径的默认值策略。

### 场景 2：航线自动规划与可视化编辑

**用户话术示例**：
- "用 D-12 给这几个图斑规划航线" / "规划一条能覆盖这些图斑的航线"
- "我手动调整一下航线" / "3 号航点往西挪一点"（后半句在编辑器里完成）

**Agent 执行链路**：
1. 调 `generate_route`（多图斑覆盖策略）→ 航线落图，覆盖图斑高亮
2. 用户要求调整 → 调 `open_route_editor` 拿免登录 URL → 右栏 iframe 切入编辑界面
3. 用户编辑完成 → 编辑页 postMessage 回传（**不走 MCP**，走前端 AG-UI 事件）→ Agent 调 `get_route_detail` 拿最新版本，向用户复述变更影响（航程/时长变化）

**需封装的 MCP tool**（route-planning-mcp）：

| tool | 入参 | 返回 | 对接系统 |
|---|---|---|---|
| `generate_route` | drone_id, plot_ids[], strategy（single/multi_cover，默认 multi_cover）, altitude_m?, overlap_rate? | route_id、版本号、航程 km、预计时长、覆盖图斑清单、GeoJSON 航线 | 航线规划算法服务 |
| `get_route_detail` | route_id, version? | 航线全量：航点列表、各段高度/速度、覆盖分析、与上一版本 diff | 航线库 |
| `open_route_editor` | route_id | 免登录 URL（临时 token，有效期 10min）、postMessage 通道约定 | 航线编辑前端 + 鉴权服务 |

**右栏联动**：`show_map`（航线虚线+覆盖高亮）→ `show_iframe`（编辑器）→ 回 `show_map`（rev.2 航线）。
**验收标准**：编辑闭环（打开→编辑→回传→Agent 复述变更）全程无人工干预链路；token 过期有兜底提示。
**难点**：免登录 token 的签发与权限收敛（只开放该 route 的编辑权）；postMessage 事件与 Agent 会话的关联。

### 场景 3：航线逻辑的自然语言解释（演示创新亮点）

**用户话术示例**：
- "为什么这么规划？" / "这条航线是怎么考虑的？"
- （Agent 也应在生成航线后主动解释，不等用户问）

**Agent 执行链路**：
1. 调 `explain_route` → 拿到**结构化决策数据**（不是文本）
2. LLM 把结构化数据转成自然语言（"你本次只需核查 GM-04，但 GM-02、GM-03 恰好在同一航向带上，一条航线顺带覆盖，比分三次起飞节省约 40 分钟"）

**需封装的 MCP tool**（route-planning-mcp）：

| tool | 入参 | 返回 | 对接系统 |
|---|---|---|---|
| `explain_route` | route_id | 结构化决策依据：覆盖图斑及各自覆盖率、航向带合并原因、避开的障碍/禁飞要素、与单图斑方案的航程/时长对比 | 航线规划算法服务（**需算法侧改造吐决策数据，D1 排期，全项目唯一硬依赖**） |

**右栏联动**：`show_map` 上高亮被顺带覆盖的图斑。
**验收标准**：解释内容与算法实际决策一致（不允许 LLM 编造理由）；降级路径：算法侧来不及时，LLM 基于航线 GeoJSON + 图斑位置直接推理解释（文档标注为推测性解释）。
**难点**：算法侧数据结构定义——建议最小集：`covered_plots[]（含覆盖率）`、`merge_reason`、`avoided_features[]`、`baseline_comparison`。

### 场景 4：飞前智能检查（CoT 展示）

**用户话术示例**：
- "我要起飞" / "可以飞了吗？" / "起飞前帮我检查一下"

**Agent 执行链路**：
1. 依次调 5 个检查 tool（也可调聚合 tool，见下），每项结果流式渲染到左栏 CoT + 右栏报告卡片
2. LLM 汇总结论（含"注意"项的解释，如空域许可到期时间）
3. 弹出 🔒 起飞确认卡片 → 人工确认 → 调 `take_off`

**需封装的 MCP tool**（preflight-mcp + flight-task-mcp）：

| tool | 入参 | 返回 | 对接系统 |
|---|---|---|---|
| `check_weather` | location, time_window? | 风速/能见度/降水/温度 + 适飞结论 | 气象服务 API |
| `check_battery` | drone_id, route_id | 当前电量、预计续航 vs 任务时长、余量结论 | 大疆上云 API + 航线库 |
| `check_route_obstacle` | route_id | 仿地飞行开关、航线净空分析、避开要素清单 | 航线规划算法服务 |
| `check_drone_obstacle` | drone_id | 机载视觉/雷达避障自检状态 | 大疆上云 API |
| `check_airspace` | route_id, time_window | 空域许可有效期、冲突活动提醒 | 空域管理系统 |
| `preflight_check`（聚合） | drone_id, route_id | 上述五项一次返回（供快速链路用；演示用单项串调以展示 CoT） | 以上各项 |
| `take_off` 🔒 | drone_id, route_id, confirm_token（确认卡片签发） | flight_task_id | 大疆上云 API / 机场 Dock |

**右栏联动**：`show_report`（五项逐条出现，通过/注意 pill）→ 确认后回 `show_map` 进入飞行态。
**验收标准**：五项检查全链路 ≤ 15s；**无 confirm_token 时 take_off 必须拒绝执行**（安全红线测试用例）。
**难点**：各子系统状态接口是否齐全（D1 盘点）；confirm_token 的签发与一次性校验。

## P1：演示后 1~2 个月（生产化第一批）

### 场景 5：飞行任务全程监控与异常播报

**形态**：Agent 从"被动问答"转为"主动执行"——任务期间持续订阅遥测，异常时**主动**在对话流里播报并给处置建议；用户也可随时问"现在飞到哪了"。

**话术示例**："现在任务进度怎么样？" / （Agent 主动）"D-12 电量降至 25%，低于返航阈值预留，建议提前收尾最后一个图斑后返航，是否执行？"

**需封装的 MCP tool**（flight-task-mcp）：

| tool | 入参 | 返回 | 说明 |
|---|---|---|---|
| `get_task_status` | flight_task_id | 进度%、当前位置、已覆盖图斑、电量、遥测摘要 | 轮询用 |
| `subscribe_task_events` | flight_task_id, event_types[]（低电量/偏航/失联/避障触发） | 订阅确认；事件经后端推送唤醒 Agent | MQTT → 事件总线 → Agent 回调 |
| `return_home` 🔒 | drone_id, reason | 指令回执 | 高危写操作 |
| `pause_task` / `resume_task` 🔒 | flight_task_id | 指令回执 | 高危写操作 |

**右栏联动**：`show_stream`（直播流 + 遥测面板）；地图升级 Cesium 3D 在此场景兑现价值。

### 场景 6：作业成果自动报告

**话术示例**："把这次任务的报告生成一下" / （任务结束 Agent 主动）"任务完成，核查报告草稿已生成，需要我读一下要点吗？"

**需封装的 MCP tool**（flight-task-mcp + data-query-mcp）：

| tool | 入参 | 返回 | 说明 |
|---|---|---|---|
| `get_task_results` | flight_task_id | 影像数量、点云、覆盖图斑、飞行统计、成果入库地址 | 成果库 |
| `generate_task_report` | flight_task_id, template?（核查报告/巡检日报） | report_id、报告结构化内容 + 可下载 URL | 报告服务（长任务，拆提交+查询） |

### 场景 7：历史数据自然语言查询（日活入口，最易做）

**话术示例**："上个月光明区飞了几架次？" / "GM-03 历史上核查过几次，结论分别是什么？" / "哪些图斑到现在还没飞过？"

**需封装的 MCP tool**（data-query-mcp）：

| tool | 入参 | 返回 | 说明 |
|---|---|---|---|
| `query_flight_history` | region?, date_range?, drone_id?, plot_id? | 架次列表 + 统计摘要 | 只读，风险最低，**建议最先上生产** |
| `query_plot_timeline` | plot_id | 该图斑全部核查记录时间线（飞行/解译/结论） | 跨库聚合 |
| `query_coverage_gaps` | region, date_range | 未覆盖图斑清单 | 支撑"还有哪些没飞"类问题 |

### 场景 8：批量任务编排（Plan-and-Execute 验证）

**话术示例**："把这 10 个图斑按优先级排期，本周之内全部飞完，每天不超过 3 架次。"

**Agent 执行链路**：LLM 生成排期计划（Plan）→ 用户确认计划 🔒 → 逐日循环调 `generate_route` + `dispatch_drone` 执行（Execute）→ 每完成一批汇报。这是**工作图（循环+分支）能力的首个真实验证**，为二期多智能体铺路。

**需封装的 MCP tool**：复用场景 1/2/4 全部 tool，新增：

| tool | 入参 | 返回 | 说明 |
|---|---|---|---|
| `create_task_plan` 🔒 | plot_ids[], constraints（截止时间/每日架次上限/优先级规则） | plan_id、逐日排期表 | 计划本身也要人工确认后生效 |
| `get_plan_progress` | plan_id | 各批次执行状态 | |

## P2：二期（3~6 个月）

### 场景 9：遥感解译 Agent（升级多智能体的触发点）

**话术示例**："把这次飞的影像解译一下" / "这次影像里有没有新增违建？" / "GM-03 和上一期比有什么变化？"

**需封装的 MCP tool**（rs-interpret-mcp，新业务域新 server）：

| tool | 入参 | 返回 | 说明 |
|---|---|---|---|
| `submit_interpret_task` | image_batch_id 或 flight_task_id, model_type（变化检测/地物分类/目标识别） | interpret_task_id | 长任务 |
| `get_interpret_status` | interpret_task_id | 进度、预计剩余时间 | |
| `query_interpret_results` | interpret_task_id 或 plot_id | 变化图斑 GeoJSON、置信度、类型 | 结果落地图 |
| `compare_plot_periods` | plot_id, period_a, period_b | 两期对比：变化区域、面积、类型 | 支撑"和上期比"类问题 |

**架构动作**：此时无人机 Agent + 遥感解译 Agent 组成 Supervisor 下双 Worker——LangGraph Supervisor 升级（届时评估 DeerFlow 作底座）在此兑现。

### 场景 10：图斑变化闭环（终局故事）

**链路**：解译发现变化（场景 9）→ Agent 自动生成复飞建议 → 人工审批 🔒 → 派飞（复用场景 1-4 全链路）→ 复飞成果再解译 → 闭环归档。

**需封装的 MCP tool**：复用为主，新增：

| tool | 入参 | 返回 | 说明 |
|---|---|---|---|
| `suggest_reflight` | interpret_task_id, threshold（变化置信度阈值） | 建议复飞图斑清单 + 理由 | 只读建议，派飞仍走 🔒 链路 |

## 长期演进路径（与录音 7.3 对齐）

```
阶段一：无人机智能体（P0+P1，单 Agent Loop → 打磨 15+12 个 tool 与评测集）
    ↓
阶段二：遥感解译智能体（P2，新增 rs-interpret-mcp，升级 LangGraph Supervisor 双 Worker）
    ↓
阶段三：其他业务按同一方式复制（每个业务域 = 一组 MCP server + 一个 Worker Agent）
    ↓
终态：Agent 自由组装、自由编排、自由执行的运行底座
```
