# 技术实现路线推荐

**日期**：2026-07-10
**依据**：《无人机项目智能体重构方案文档》（7月10日内部讨论录音）+ 2026年中业界智能体方案调研

---

## 一、一句话路线

**用 MCP 把无人机业务能力封装成原子化工具层；智能体"两步走"——演示期用单 Agent Loop（能出外网选 Claude Agent SDK；内网/国产选 LangGraph 最小图 + Qwen），生产期长成 LangGraph Supervisor 多智能体（DeerFlow 作参考架构），模型用私有化 Qwen/GLM；前端 CopilotKit（AG-UI）分屏界面；放飞类高危操作强制人在环审批。**

对应录音三个核心诉求：

| 录音诉求 | 技术承接 |
|---|---|
| 业务原子化、Agent 自由组装 | MCP 工具层（能力 = MCP tool，非 API 清单） |
| Agent Loop / 工作图，非 Dify 线性工作流 | LangGraph 图编排（循环、条件分支、checkpoint） |
| Codex Cloud Cowork 式层级（沙Agent/业务Skill） | Supervisor-Worker 多智能体 + Skill 化工具集 |

---

## 二、工具层：MCP 为主，CLI 为辅

录音中"CLI or MCP"之争的答案：**MCP 为主、CLI 为辅**。

**选 MCP 的理由（2026年中状态）：**
- 已是事实标准：Linux Foundation 中立托管，官方 Registry 近万个 server，全生态约 2 万个。
- 2026-07-28 新版带 **Tasks 长时任务扩展**——正好匹配无人机分钟~小时级飞行任务；另有无状态核心（可水平扩展）、MCP Apps（服务端渲染 UI）。
- 私有化基础设施成熟：**Higress（AI 网关）+ Nacos MCP Registry** 可自建纯内网 MCP 市场。
- 换模型/换编排框架时工具层原封不动——这是"两步走"能成立的前提。

**CLI 的位置：** 开发期快速验证。把能力先包成命令行跑通，再平移为 MCP tool，二者共用同一层业务原子化代码。开发测试阶段可用 **pi**（极简 agent harness，MIT，OpenClaw 的引擎）驱动这些 CLI 跑真实话术，快速暴露参数设计和能力缺口——承接录音"用智能体测、反向驱动代码完善"的方法论。

**MCP server 拆分（按业务域拆 server、按原子能力拆 tool）：**

一个 tool 的粒度标准：一句自然语言能说清它干什么，且参数可由上下文推断。

```
drone-dispatch-mcp（调度域）
  query_plots(region?, plot_type?, date_range?)        图斑查询
  find_nearby_drones(plot_id | location, radius_km)    周边无人机查询
  dispatch_drone(drone_id, task_type)                  无人机调度（人在环确认）

route-planning-mcp（航线域）
  generate_route(plot_ids[], coverage_strategy)        航线生成（多图斑覆盖）
  explain_route(route_id)                              返回结构化决策依据 → LLM 转自然语言
  open_route_editor(route_id) → url                    免登录编辑界面地址

preflight-mcp（飞前检查域）
  preflight_check(drone_id, route_id)                  聚合：天气/电量/避障/空域
```

长时任务模式：工具立即返回 `task_id`，Agent 轮询 `get_task_status` 或事件回调唤醒（MCP Tasks 扩展落地前用"提交-查询"两个 tool 模拟）。

设备侧：**大疆上云 API**（MQTT 物模型、支持私有化）封装为 MCP server 即可让 Agent 调度机场/机队。行业背书：大疆 FlightHub 2 已官方内置 AI Copilot（自然语言下发任务、VLM 参与航线规划）——方向已被头部厂商生产验证；我们的差异化是打通"图斑→调度→航线→检查→解译"全业务链。

## 三、智能体选型：两步走

### 3.1 演示期（至 7.22）：单 Agent Loop

**为什么单 Agent 不是多智能体：**
- 演示四场景是一条串行工具链，一个 ReAct 循环就能走通，无并行/隔离需求。
- 业界共识（Anthropic/Cognition）：先单 Agent + 好工具，确有需求再拆。多智能体的协调失败与"通信幻觉"级联正是录音第八章警告的头号风险（40% 智能体项目死于过度编排）。
- 现场原则：减少故障点。单 Agent 出错只有一处可查。

**框架二选一：**

| | Claude Agent SDK | LangGraph 最小图 + Qwen 模型 |
|---|---|---|
| 定位 | 技术最优解 | 合规最优解 |
| Agent Loop | 开箱即带，零框架代码 | 单节点 ReAct 循环 + MCP 工具，几十行代码 |
| MCP 集成 | 业界最深（一行接入+进程内 server） | langchain-mcp 适配，成熟 |
| 前端对接（CopilotKit/AG-UI） | 不走 AG-UI，需自发 5 种 SSE 事件（演示期够用） | **AG-UI 一等公民，零胶水** |
| 代码开源 | SDK 开源（MIT），运行时+模型闭源 | 全栈开源（LangGraph MIT + Qwen 开源权重） |
| 纯内网运行 | ❌（仅 Anthropic/Bedrock/Vertex 云 API） | ✅（vLLM 私有化） |
| 加分项 | Subagents/Skills/Hooks 与"沙Agent/业务Skill"同构；放飞确认可做成 PreToolUse hook | **与生产期架构同源，演示代码直接长成 Supervisor，零迁移** |

**决策规则：演示环境能出外网 → Claude Agent SDK（最快出效果）；客户要求内网/国产化 → LangGraph 最小图 + Qwen 一条路走到底。** 工具层是 MCP 标准协议，二者切换成本接近零。

> **关于 Qwen-Agent 的定位修正**：AG-UI 官方集成列表（LangGraph、CrewAI、Mastra、PydanticAI、AG2、微软 Agent Framework、AWS Bedrock 等）**不含 Qwen-Agent**，接 CopilotKit 前端需自写 200-400 行适配层（可参照 AG2 集成的实现）。因此国产路线的产品链路选 LangGraph + Qwen 模型而非 Qwen-Agent；**Qwen-Agent 降级为开发期快速验证 MCP 工具的内部工具，不进产品链路**。

### 3.2 生产期：LangGraph Supervisor 多智能体

- **LangGraph 1.x**（2025-10 发 1.0，生产级稳定）：图编排最强——循环/条件分支/持久 checkpoint（PostgreSQL）/**human-in-the-loop 中断恢复**（放飞审批的原生支撑）/time-travel 调试；官方 supervisor/swarm 模式库；模型无关。
- 结构：Supervisor + 无人机 Agent（调度/航线/飞控）+ 遥感解译 Agent（二期）。
- **DeerFlow（字节开源，基于 LangGraph 1.0，2026-02 发 2.0，月内近 4.9 万 star）作为参考架构**：其 Leader-Worker 结构、Markdown Skill 系统、Docker 沙箱、分层记忆与我们目标高度同构。二期评估两条路：① LangGraph 自建 Supervisor（干净）；② fork DeerFlow 换掉其子 Agent（白捡工程件）。注意其项目仅 5 个月大、roadmap 字节主导，绑定需谨慎——好在底座是 LangGraph，退路干净。
- 已知短板应对：LangGraph checkpoint ≠ 真正 durable execution，副作用幂等自己管；派飞类写操作全部做幂等键。

### 3.3 模型选型（国产私有化）

| 模型 | 评估 | 结论 |
|---|---|---|
| **Qwen3.x** | 工具调用一次正确率 ~94%，百炼原生 MCP，Qwen-Agent 配套 | 首选 |
| **GLM 5.x** | 面向 agent 设计，有政府审批资质，政企合规最好 | 并列首选 |
| DeepSeek | 老模型 2026-07-24 弃用换 v4；strict function calling 仍 Beta 有已知 bug；三家中工具调用最弱 | 备选 |

三家均暴露 OpenAI 兼容 function calling；vLLM/SGLang 私有化均带 tool-call 解析模板。**国产模型 + 自建 MCP 网关的纯内网 Agent 栈在 2026 年中已完全可行。**

## 四、被排除的选项及理由

| 选项 | 排除理由 | 保留价值 |
|---|---|---|
| **Dify / Coze / n8n** | 线性工作流（录音已否定）；上下文管理/重试/成本核算被抽象掉，生产口碑差；OpenAI AgentKit 已宣布 2026-11 下线，低代码路线风险实证 | 仅作演示 L3 兜底 |
| **OpenAI Agents SDK** | 深度绑定 OpenAI 平台，国内政企不可用 | 无 |
| **pi** | 定位错配：给开发者的终端编码 agent，刻意不做 MCP/子代理/权限确认；不可嵌入业务前端 | 开发期测试 harness；其"克制复杂度"哲学内化为架构纪律 |
| **Qwen-Agent** | 无 AG-UI 官方集成，接 CopilotKit 前端需自写适配层；能力被 LangGraph+Qwen 组合覆盖 | 开发期后端验证 MCP 工具的内部工具 |
| **A2A 协议** | agent↔agent 跨组织协作，当前阶段用不上 | 三期跨部门智能体协作预留接口 |

## 五、多智能体工程纪律（直接采纳为设计约束）

1. 先单 Agent，确有并行/隔离需求再拆。
2. Supervisor/Manager-Worker 是生产默认模式；Handoff 设移交次数上限防无限循环。
3. 子代理窄职责 + 受限工具集；**子代理返回结构化 JSON 而非自由文本**。
4. 每个角色用最便宜的胜任模型（意图理解用大模型，参数抽取/格式化可用小模型）。
5. 有意限制复杂性（录音共识 + pi/OpenClaw 哲学）。

---

## 六、安全治理（生产必做，演示声明即可）

- MCP 网关（Higress）统一鉴权、审计日志、工具白名单——防提示注入越权调用。
- 高危操作（起飞/返航/急停）双保险：Agent 侧人在环确认 + 业务侧权限校验；**Agent 的 token 权限 ≤ 当前登录用户权限**。
- 全部 Agent 决策留痕（LangGraph checkpoint 天然提供），满足政企审计。
