## 二、降级方案（三级，比"写死 Token"更稳）

| 级别 | 触发条件 | 做法 | 观感损失 |
|---|---|---|---|
| L1 | 个别 tool 不稳 | 该 tool 内部走写死参数的胶水代码，Agent 链路不变 | 无 |
| L2 | 意图理解不稳 | 演示话术收敛为固定 5 句，对这 5 句做关键词兜底路由直配工具链 | 无（话术照稿念） |
| L3 | Agent Loop 整体不可用 | Dify 搭线性工作流版本保底，界面壳一致 | 失去自由对话，流程可演 |

## 三、测试方法论：智能体测试反向驱动（录音方法论落地）

1. **工程层**：每个 MCP tool 做自动化契约测试（参数 schema 校验、返回结构断言）。
2. **Agent 层**：建**评测集**——30~50 条真实业务话术（含模糊表达、口语化表达），跑批统计三个指标：**工具命中率、传参正确率、任务完成率**。
3. **调优顺序**：命中率低的 tool → 先改 name/description/参数说明（**工具描述是 Agent 的"文档"，是命中率第一杠杆**）→ 仍不行才改代码。
4. **开发期辅助**：业务能力先包成 CLI，用 pi 驱动 CLI 跑话术，快速暴露参数设计与能力缺口，再平移为 MCP tool；MCP tool 的后端验证可用 Qwen-Agent 快速起对话环境（仅内部工具，不进产品链路）。

**开发者新工作形态**（对应录音"你以后不是写代码了"）：
- 写 tool description 代替写胶水代码
- 维护评测集代替手工点测（每次改动看命中率/完成率曲线）
- 看 Agent 执行轨迹代替看日志（LangGraph time-travel / SDK transcript 即新调试器）
- 能力缺口清单驱动排期：缺 tool 补 tool、缺描述改描述、缺代码才写代码

## 六、安全红线（不可妥协项）

1. `dispatch_drone` / `take_off` / 返航 / 急停：Agent 只能生成待确认指令，**人点击确认后才执行**（演示期 = 确认卡片；生产期 = LangGraph interrupt + 业务侧权限二次校验）。
2. Agent 的 token 权限 ≤ 当前登录用户权限。
3. 全部 Agent 决策留痕可审计。
