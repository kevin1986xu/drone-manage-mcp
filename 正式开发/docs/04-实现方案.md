## 一、总原则：零 fork、插件式集成

**DeerFlow 作为依赖使用，绝不改它的核心代码。** 依据（读代码确认）：它的扩展点全部是
"import path 注入"式——`mcpInterceptors`（extensions_config 里写 `my_package.mod:factory`）、
`custom_agents`（config.yaml 声明子代理）、`memory.manager_class`、`sandbox.use`、
skills 外挂目录。我们的全部定制放**自己的 Python 包 + 配置 + skills 目录**里，
DeerFlow 升级时只跟 CHANGELOG 迁移配置，不解决代码冲突。

## 二、正式版目录结构

```
正式开发/
├── deploy/                     # DeerFlow 部署配置（不含其源码）
│   ├── config.yaml             # 模型（qwen→生产 vLLM/MindIE）、子代理、沙箱、记忆
│   ├── extensions_config.json  # MCP servers（核心四件套直连/同步桥维护）+ interceptors + skills 注册
│   └── docker-compose.yml      # deerflow + nginx + 我们的服务
├── uav_extensions/             # 我们的 Python 包（pip 安装进 DeerFlow 运行环境）
│   ├── interceptors/           #   API key 注入 + 工具调用审计日志（mcpInterceptors 注入点）
│   ├── nacos_bridge/           #   Nacos MCP Registry → PUT /api/mcp/config 同步桥（含防抖/对账）
│   └── approval/               #   高危审批服务（confirm_token 签发/校验 REST，独立小模块）
├── mcp-services/               # FastMCP 业务服务（自演示版平移 + API key 校验）
│   ├── drone_dispatch / route_planning / preflight / flight_task
│   └── recognition/            #   智能识别域（新增，对接识别算法平台）
├── skills/                     # 业务 SKILL.md → 挂载为 DeerFlow 的 skills/custom
│   ├── plot-inspection/        #   图斑核查派飞全流程（查→选机→规划→检查→审批起飞）
│   ├── batch-patrol/           #   批量巡查排期
│   └── smart-recognition/      #   智能识别编排（起飞采集→识别→结果落图）
├── frontend/                   # GIS 态势前端（自演示版 AG-UI 分屏演进）

```

## 三、关键实现方案

### 3.1 高危审批（最重要的设计）

**原则不变：token 签发在 Agent 之外，Agent 永远拿不到自批权。**

```
工具层（不变）：take_off 无 token → 返回 requires_confirmation + action_id（自拒）
                       │
审批服务（uav_extensions/approval，独立 REST）：
    人工动作 → POST /approve/{action_id} → 签发一次性 confirm_token
                       │
确认入口（按对话渠道适配）：
  · GIS 前端：确认卡片（演示版组件平移）→ 调审批服务 → 把 [SYSTEM_CONFIRMATION]
    经 DeerFlow runs API 投递回该 thread（与演示版机制同构）
  · 企微/钉钉：**交互式卡片按钮**（两平台原生支持）→ 回调审批服务 → 同上投递
  · 兜底：ask_clarification 只用于"提示用户去点卡片"，绝不把用户文本"确认"
    当作授权（文本可被诱导，按钮+服务端签发不可）
```

加一层保险：`uav_extensions/interceptors` 里做**硬白名单拦截**——take_off/publish 类
工具调用若无有效 token 格式，拦截器直接短路拒绝（纵深防御，防提示注入绕过提示词）。

### 3.2 核心业务域接入（同步桥）

`nacos_bridge`：订阅/轮询 Nacos v3 MCP Registry → diff → `PUT /api/mcp/config` 热更新
→ 触发工具缓存重载。带防抖（注册风暴不抖动 DeerFlow）与启动对账（全量校正）。
核心四件套由桥维护成"直见工具"；nacos-mcp-router（router 模式）另配一条，只服务长尾发现。
**桥做完先在我们仓库用，稳定后提 PR 回 DeerFlow 上游。**

### 3.3 GIS 态势前端（复用演示版，BFF 适配）

保留 AG-UI 分屏（地图/航线编辑/排期表/确认卡片都是现成的），新增薄 BFF：
订阅 DeerFlow run 的 SSE（LangGraph 兼容事件）→ 复用演示版 `llm_runner` 的映射逻辑
（tool 结果 → show_map/show_report/show_confirm 视图指令）→ 前端零改动消费。
这层适配与演示版 `astream_events → AG-UI` 是同构问题，逻辑可直接抄。

### 3.4 子代理划分（窄职责 + 受限工具集）

| 子代理 | 工具集 | 说明 |
|---|---|---|
| flight-ops（飞控） | 调度/航线/飞前/飞行任务四域 | 唯一能碰高危工具的代理 |
| recognition（识别） | recognition 域 MCP | 只读+提交识别任务，无飞控权限 |
| lead-agent | 编排 + skills | 不直接持有高危工具 |

工具集边界写进 config.yaml 子代理定义——"识别代理永远调不到 take_off"是配置保证的，
不靠提示词。

### 3.5 模型与部署

- 开发期：qwen3.7-max（现有 token-plan 端点，OpenAI 兼容直配）
- 生产：vLLM（Qwen/GLM）或 MindIE（昇腾）——DeerFlow 模型工厂原生支持，改 config.yaml
- 部署：docker-compose 起 DeerFlow 全家桶 + 我们的 mcp-services/审批服务/BFF；
  Nacos 沿用现网 3.x

## 四、里程碑（建议节奏）

| 阶段 | 内容 | 量级 |
|---|---|---|
| M0 POC | 03 文档三个硬指标（链路/审批体验/延迟） | 1~2 天 |
| M1 骨架 | 目录落地、DeerFlow 部署、四件套直连、API key 鉴权、审批服务 + GIS 前端确认卡片打通 | ~1 周 |
| M2 能力 | 三个业务 SKILL.md、子代理划分、Nacos 同步桥、评测集移植跑绿 | 1~2 周 |
| M3 前端 | BFF 适配 + GIS 分屏完整迁移（可与 M2 并行） | ~2 周 |
| M4 扩展 | 识别域 MCP + recognition 子代理、企微/钉钉卡片审批、遥测主动播报 | 按需排期 |

每个里程碑的验收都跑评测集 + 端到端浏览器实测（沿用演示版的验证纪律）。

## 五、为什么不推荐另外两条路（简记）

- **fork DeerFlow 改核心**：升级即冲突，577+ issue 的年轻项目跟不动；它的扩展点足够，没必要
- **继续纯自研加深**：单业务嵌入式够用，但你的目标是平台（多场景/多入口/能力沉淀），
  平台件（入口/鉴权/记忆/定时/skill 管理）自研成本远超集成成本
