# 07 - 对外服务化与鉴权架构

> 回答三个问题：**① 这套系统里哪些是对外产品、哪些只是验证脚手架；
> ② 如何分层、如何对外提供；③ 现在裸奔的接口如何加鉴权、如何安全地给第三方用。**
> 承接 [02 架构定稿](02-架构定稿与建议.md) §2.4「多消费方时插 Higress」的预留，
> 和 [05 扩展规划](05-MCP工具与Skills扩展规划.md) §4.3「Agent 权限 ≤ 用户权限」。
> 日期：2026-07-20。

## 0. 一句话结论

**对外交付的只有两样：MCP 能力（8 域工具）+ 人在环安全模型（confirm_token）。**
DeerFlow、演示版后端、GIS 前端、BFF、评测集全部是**验证脚手架**——它们证明这套
MCP 能力可用、好用、安全，但不进对外产品边界。对外用户拿到的是「一组带鉴权的
MCP server + 一本能力手册」，他们用自己的 Agent（Claude / Cursor / 自研）接入，
或用我方托管的 DeerFlow。

鉴权现状：**三道关只有中间一道是成熟的**——接入关是单一共享密钥（可绕过、无租户、
无限流），操作关 confirm_token 成熟，回源关（调 drone-manage 平台）**完全裸奔**
（无任何身份，等于超级管理员）。本文给出三道关的补齐方案与分阶段路线。

## 1. 产品边界：对外的只有 MCP + Skills

| 组件 | 定位 | 对外？ |
|---|---|---|
| **mcp-services（8 域工具）** | **产品·能力层** | ✅ 核心交付物 |
| **审批服务（confirm_token 签发）** | **产品·安全配套** | ✅ 随能力层交付（高危动作的授权中心） |
| **Skills（6 个 SKILL.md）** | **产品·编排层** | ◐ 可选——消费方用 DeerFlow/Claude 技能生态才需要；自带 Agent 的消费方不需要 |
| DeerFlow（Gateway + Web） | 宿主·我方参考实现 | ✗ 我方自用，不强绑给对方 |
| 演示版 backend（8000）+ GIS 前端 | 验证·自证能力 | ✗ 演示与验收专用 |
| BFF（8300） | 验证·前端适配层 | ✗ 演示版前端接 DeerFlow 的胶水 |
| eval（60 条）+ contract_smoke | 验证·质量门禁 | ✗ 内部 CI/演示前门禁 |

**判断依据**：MCP 是协议标准，任何 MCP 客户端都能接；Skills 是 DeerFlow/Claude 的
编排格式，跨生态不通用。所以对外的硬边界是 MCP server，Skills 是「我方最佳实践的
可选赠品」。别把 DeerFlow 当产品对外——它是我们自己选的宿主，对方可以换。

## 2. 分层架构与信任边界

```
                          ┌─────────────── 消费方（多租户）───────────────┐
                          │ 我方 DeerFlow │ 第三方 Agent │ 合作方系统/大屏  │
                          └───────┬───────────────┬──────────────┬────────┘
   ── 信任边界①（对外）──────────┼───────────────┼──────────────┼──────
                          ┌───────▼───────────────▼──────────────▼────────┐
   L3 接入层              │   API 网关（Higress）：鉴权·限流·审计·多租户    │  ← 对外唯一入口
                          └───────────────────────┬────────────────────────┘
   ── 信任边界②（内网，仅收网关流量）─────────────┼──────────────────────
                          ┌───────────────────────▼────────────────────────┐
   L2 编排层（可选）      │   Skills（SKILL.md 工作流）· 运行在消费方 Agent 内 │
   L1 能力层             │   mcp-services 8 域（无状态工具）+ 审批服务       │
                          └───────────────────────┬────────────────────────┘
   ── 信任边界③（回源，最大缺口）─────────────────┼──────────────────────
                          ┌───────────────────────▼────────────────────────┐
   L0 领域层             │   drone-manage 平台（若依）· 真相源 · dataScope   │
                          └─────────────────────────────────────────────────┘
```

- **信任边界①（对外）**：消费方身份在此确认。当前只有单一 X-API-Key，是本次要重点补的关。
- **信任边界②（内网）**：网关到工具面。当前 mcp-services 直接监听 0.0.0.0，**任何人
  直连 820x 端口即绕过网关**——要用网络策略/mTLS 收口为「只收网关流量」。
- **信任边界③（回源）**：工具面到平台。**当前完全无鉴权**（`DroneManageClient._call`
  裸发 HTTP），平台无条件信任 mcp-services，等于所有 Agent 操作都是超管权限。

## 3. 对外提供的三种模式

| 模式 | 适用消费方 | 说明 |
|---|---|---|
| **A. MCP 直连（streamable-http）** | 任何标准 MCP 客户端（Claude Desktop/Code、Cursor、自研 Agent） | 最标准。对方配置 `url + 鉴权头` 即可 `tools/list` 自发现。我方出能力手册说明语义与风险分级 |
| **B. 经网关的 MCP（生产推荐）** | 同上，但需多租户治理 | 网关做租户鉴权+限流+审计，后端 server 不变。对方拿到的是网关地址而非直连端口 |
| **C. Skills 打包** | 也用 DeerFlow / Claude 技能生态的消费方 | 附赠 6 个 SKILL.md，对方直接获得成熟编排（图斑核查/排期/值班等） |

**对外契约**：MCP 的 `tools/list` 是机器可读的自描述（工具名/入参 schema/描述），
再配一份人读的能力手册（§6）。两者足以让第三方零沟通接入。

## 4. 鉴权体系：三道关

### 4.0 现状盘点

| 边界 | 现状 | 缺口 | 进展（2026-07-20） |
|---|---|---|---|
| ① 消费方 → 工具面 | `ApiKeyMiddleware` 校验单一 `X-API-Key`（hmac 比对） | 单密钥无法区分租户；无限流/配额；直连端口即绕过 | **多租户 key 已实现**（`UAV_TENANT_KEYS`，key→租户，注入 `scope.uav_tenant`，向后兼容单 key）；限流/白名单待网关 |
| 操作级 → 高危动作 | `confirm_token`（审批服务签发，一次性/绑定动作/TTL/防重放/防伪造） | ✅ 成熟，保持 | — |
| ③ 工具面 → 平台回源 | `DroneManageClient._call` **裸 HTTP，无任何身份** | 无用户身份、无 dataScope、Agent = 超管；越权与审计追责均无从谈起 | **注入框架已实现**（`_auth_headers`：服务账号 token + contextvar 用户透传，不配则裸调）；待平台给账号 + 平台侧认身份 |

### 4.1 关一 · 接入鉴权（消费方是谁）

**阶段 1（快，1-2 天）：多租户 API Key。**
把 `UAV_MCP_API_KEY` 单密钥升级为「key → 租户」映射，每租户带元数据：
- 工具白名单（如只读租户拿不到任何 🔒 高危工具）；
- 速率/配额；
- 回源身份（映射到平台的服务账号或用户，见关三）。

落点：`auth.py` 的 `ApiKeyMiddleware` 改为查表（key→tenant），把 tenant 注入
request 上下文供拦截器与审计用。密钥用 `Authorization: Bearer <key>` 或保留
`X-API-Key`，值改为「租户前缀+随机」便于溯源。

**阶段 2（成熟）：OAuth 2.1 Resource Server（MCP 官方授权规范）。**
MCP 2025-06 授权规范把 MCP server 定义为 **OAuth 2.1 Resource Server**：
- server 暴露 **Protected Resource Metadata（RFC 9728）**，声明其授权服务器地址；
- 客户端走 **Authorization Code + PKCE** 到授权服务器换 access token；
- token 带 **Resource Indicators（RFC 8707）** 绑定 audience，防止被重放到别的 server；
- 网关/server 用授权服务器的 JWKS 验签 token，读 scope 决定工具可见性。

授权服务器可选：对接**若依自带的用户体系**（政务客户已有账号）、企业 IdP
（Keycloak/Authing），或平台若已上 Sa-Token/OAuth 直接复用。这一步让标准 MCP
客户端（Claude/Cursor）零摩擦接入，是对外生态就绪的标志。

### 4.2 关二 · 操作授权（能不能执行这个高危动作）

**confirm_token，已成熟，原样保持。** 与关一是正交的两层：
关一确认「你是谁、能看哪些工具」，关二确认「这一次高危动作由现场的人点头」。
12 个 🔒 高危工具 + guard 拦截器伪造 token 拦截已实测。两层不可互相替代。

### 4.3 关三 · 回源授权（以谁的身份读写平台）——最大缺口

现在 mcp-services 裸调平台 = 所有操作都是超管，**越权无边界、审计无主体**。
政务客户的合规必答题就卡在这。补齐分两步：

**先接一个真实平台账号（阶段 1，止血）**：`DroneManageClient` 增加平台登录/token
注入，`_call` 默认带 `Authorization`，别再裸调。至少让平台侧知道「是 Agent 服务
账号干的」，审计有主体。

**再做用户身份透传（阶段 2，落地「Agent≤用户」）**：
```
消费方登录态 → 网关/DeerFlow 透传用户 token → mcp-services 拦截器按调用上下文
注入平台用户身份头 → drone-manage 用该用户身份做 dataScope 数据过滤 +
菜单权限前置校验（如 drone:task:publish）
```
落点：`_call(**kw)` 支持按 request 上下文注入用户头（拦截器从关一的 token 里取
用户身份）；平台侧接 OAuth/Sa-Token 认这个身份。**原则：Agent 能做的 ≤ 发起
用户在平台的权限**——读受 dataScope 过滤，高危写受菜单权限校验。这是关一 token
里的用户身份一路透到 L0 的闭环。

## 5. 网关落地：Higress（承接 02 的预留）

02 §2.4 判定「单消费方不上 Higress，多消费方时插入，DeerFlow 只改一条配置指向
网关」。**对外 = 多消费方，触发条件到了。**

| 网关承担 | 说明 |
|---|---|
| 服务发现 | 从 Nacos（现网 192.168.101.21:8998）发现 8 个 MCP server，免手维护地址 |
| 消费方鉴权 | 关一：阶段 1 校验租户 key / 阶段 2 校验 OAuth JWT（JWKS 验签） |
| 限流·配额 | 按租户；保护平台不被 Agent 打爆 |
| 审计 | 网关访问日志 + trace-id，与拦截器/平台三处留痕关联 |
| 白名单·收口 | 工具级白名单；mcp-services 用网络策略只收网关流量，堵直连绕过 |

部署位：`消费方 → Higress:8080 → mcp-services:820x`。Higress 基于 Envoy、原生
MCP 支持，且与现网 Nacos 同厂配套。备选 Kong/APISIX（若不用 Higress）。
**mcp-services 代码基本不动**——鉴权升级在网关 + `auth.py`，业务工具零改。

## 6. 对外交付物清单（让别人能接）

1. **MCP 能力手册**：8 域工具清单 + 语义 + 风险分级（读/写/🔒/⚡）+ 人在环说明
   （由 06 功能明细裁剪对外版，去掉平台内部字段）；
2. **接入端点 + 鉴权说明**：网关地址 + 如何申请 key / 走 OAuth；
3. **租户开通流程**：申请 → 分配 key/client → 工具白名单 → 配额 → 沙箱验证；
4. **沙箱租户**：`UAV_CREATE_REAL_TASK=0` 的只读/仿真租户，对方先在沙箱验证再上生产；
5. **SLA / 限流 / 审计承诺**：可用性、速率上限、留痕与追责口径。

## 7. 分阶段路线

| 阶段 | 目标 | 内容 | 依赖 |
|---|---|---|---|
| **P0 对外最小可用** | 多租户可控接入 | 关一阶段 1（多租户 key，改 `auth.py`）+ Higress 插入（鉴权/限流/审计）+ mcp-services 网络收口 + **关三止血**（回源接真实平台账号，别再裸调）+ 能力手册 + 沙箱租户 | Higress 部署、平台服务账号 |
| **P1 用户级授权** | 落地 Agent≤用户 | 关三阶段 2（身份透传 + 平台 dataScope + 菜单权限校验）；政务合规必答 | 平台侧接 OAuth/Sa-Token 认透传身份 |
| **P2 标准化生态** | 零摩擦对外 | 关一阶段 2（OAuth 2.1 Resource Server + RFC 9728/8707 + 授权服务器）；标准 MCP 客户端直接接 | 授权服务器（若依/企业 IdP） |

## 附：四条安全原则

1. **纵深防御**：每层独立校验，不靠单一边界（网关挂了，工具面的 key 校验与
   confirm_token 仍在）；
2. **最小权限**：租户工具白名单 + Agent≤用户；只读租户永远拿不到 🔒 工具；
3. **零信任回源**：平台不再无条件信任 mcp-services，回源必带可追责身份（关三）；
4. **全链路审计**：网关 + guard/audit 拦截器 + 平台三处留痕，用同一 trace-id 串联，
   高危与紧急动作可回溯到「哪个租户/哪个用户/哪次确认」。
