## 一、定稿链路

```
                    ┌────────── 对话入口：Web UI / 企微 / 钉钉 / 飞书（DeerFlow 内置通道）
                    ▼
             DeerFlow 2.0（SuperAgent 底座）
                    │
        ┌───────────┴─────────────────┐
        │ 核心业务域（固定、直见工具）    │ 长尾/新能力（动态发现）
        ▼                             ▼
  直连 or Nacos同步桥            nacos-mcp-router（router 模式）
        │                             │ search_mcp_server / use_tool
        ▼                             ▼
  FastMCP 服务(Python，streamable-http，注册进 Nacos) ◀──注册中枢── Nacos 3.x
        │
        ▼ HTTP
  无人机平台 drone-manage（Java，**零改动**）
```

**全程不引入 Higress**（决策依据见 §三、§四）。

## 二、关键决策与理由

### 2.1 为什么核心业务域不走 nacos-mcp-router 的 router 模式

nacos-mcp-router（[nacos-group/nacos-mcp-router](https://github.com/nacos-group/nacos-mcp-router)）两种模式：

| 模式 | 行为 | 适用 |
|---|---|---|
| router（默认） | 模型只看到 `search_mcp_server/add_mcp_server/use_tool` 三个元工具，先搜再经 use_tool 转调 | 海量未知 MCP 市场的动态发现 |
| proxy | 把 stdio/SSE server 转成 streamable 端点 | 协议转换（我们的 FastMCP 本身就是 streamable，无此需求） |

router 模式用在飞控主链路的代价：
1. 模型看不到 `query_plots/take_off` 等真实工具——**演示版调了三轮才到 91% 命中率的工具描述工程全部失效**；
2. 每次调用多一跳"搜→转调"，延迟与不确定性上升；
3. 高危工具经 `use_tool` 间接传参，审批链路可读性差（confirm_token 在工具内校验，安全性不丢，但排障难）。

**定稿**：核心飞控四件套（调度/航线/飞前/飞行任务）在 DeerFlow 里**直见**（直连配置或同步桥）；
router 模式作为**补充**——以后注册进 Nacos 的智能识别、新业务域，模型可自行发现使用，零配置变更。

### 2.2 核心业务域接入的两种实现（二选一，推荐 B）

- **A. 直连写死**：`extensions_config.json` 配 4 个 streamable 端点。最简，但增删 server 要改配置。
- **B. Nacos 同步桥（推荐）**：几十行服务——轮询/订阅 Nacos v3 MCP Registry → 调 DeerFlow `PUT /api/mcp/config` 热更新 + 触发工具重载（两个 API 都是现成的，已在其代码中确认）。Nacos 里注册/下线，DeerFlow 自动跟随；**该桥值得作为 PR 贡献回 DeerFlow 上游**（其当前缺注册中心发现）。

### 2.3 网关问题澄清（"DeerFlow 里不是有类似 Higress 的网关吗"）

**DeerFlow 的 Gateway(8001) ≠ Higress，两者治理的流量方向不同**：

- DeerFlow Gateway 管 **用户→Agent** 南北向：threads/runs/memory/skills/用户鉴权——是它自己的应用 API 层，不站在 Agent→工具的路径上；
- Higress 在原方案里管 **Agent→MCP 工具** 东西向：从 Nacos 发现后端、消费方鉴权、审计、白名单、限流。

DeerFlow 在工具面有**客户端侧**机制可部分替代：`mcpInterceptors`（调用前注入鉴权头/自定义拦截，可做客户端审计）、server 级 enabled 开关、运行 journal。

### 2.4 不上 Higress 的治理口径（如实边界）

单消费方（只有 DeerFlow 调这些工具）+ 内网信任前提下成立：

| 治理项 | 由谁承担 |
|---|---|
| 工具调用鉴权 | DeerFlow interceptor 注入 API key + **FastMCP 服务端校验**（需新增，工作量小） |
| 高危操作人在环 | 工具内 confirm_token（框架无关，演示版原样带走，含防伪造/防重放实测） |
| 审计留痕 | DeerFlow 运行 journal + FastMCP 服务端访问日志 + 后端决策留痕 |
| 防绕过 / 多消费方收口 / 限流 | **当前不设防**——任何人直连 FastMCP 端口即绕过客户端治理。将来需要时在工具面插入 Higress，DeerFlow 只改一条配置指向网关，链路其余不动 |

## 三、现有资产迁移映射（演示版 → 正式版）

| 演示版资产 | 去向 | 迁移成本 |
|---|---|---|
| 4 个 FastMCP server（含 Nacos 注册、drone-manage 客户端、WKT 转换、气象自查） | 原样复用（正式版的工具层就是它） | ≈0 |
| confirm_token 人在环机制 | 随工具层带走；确认交互用 ask_clarification 承载或改 DeerFlow 前端 | 小（交互适配） |
| 工具描述（三轮调优、llm 命中 91%+） | 直见接入后原样生效 | 0 |
| 业务流程（查图斑→派飞、批量排期、软约束调参） | 转写为 `skills/custom/*.md`（SKILL.md 格式与 Claude 技能同构） | 小 |
| 38 条评测集 + 跑批脚本 | 对接 DeerFlow API 重写 runner，用例复用 | 小 |
| AG-UI 分屏前端（地图/编辑器/排期表/确认卡片） | 方案①保留，对接 DeerFlow 的 LangGraph 兼容 API（SSE 事件映射与演示版 llm_runner 同构）；方案②深改其 Next.js | 中（正式版最大的一块前端工作） |
| scripted 兜底路由 | 不迁移（演示应急专用） | — |
| LangGraph 单 Agent（graph.py） | 被 DeerFlow lead-agent + 子代理取代；系统提示词的业务规范迁入 skills/子代理提示 | 小 |

## 四、需要新建的组件清单

1. **Nacos→DeerFlow 同步桥**（§2.2 B，几十行）
2. **FastMCP 服务端 API key 校验**（治理口径要求）
3. **飞控/识别子代理定义**（config.yaml custom_agents + 各自受限工具集——契合"子代理窄职责+受限工具集"纪律）
4. **业务 Skills**（图斑核查派飞、批量巡查、智能识别编排等 SKILL.md）
5. **高危确认交互适配**（ask_clarification 流或前端改造，POC 定）
6. **遥测主动播报**（MQTT→事件→对话推送，P1；可用其 message bus/channels 作载体）
7. **GIS 态势前端对接**（迁移映射表最后一行）

## 五、风险清单与应对

| 风险 | 等级 | 应对 |
|---|---|---|
| DeerFlow 无工具级审批门禁 | 高（飞控命门） | 工具内 confirm_token 兜底（已实测防伪造）；确认交互为 POC 第一验证项 |
| 单轮对话延迟（lead-agent+中间件链较重） | 中 | POC 实测；超标则精简中间件/直连模型配置 |
| 项目年轻、breaking change 快（2.0→2.1 memory 重构） | 中 | 锁定版本 + 预留升级人力；自定义物尽量放 skills/custom 与独立包，少改其核心 |
| nacos-mcp-router 项目较小（180 star） | 低 | 仅承担长尾发现，非主链路；坏了不影响核心业务域 |
| 无 Higress 的防绕过缺口 | 中（生产） | 内网边界+服务端 key 先顶；多消费方时插 Higress（链路预留位） |
| GIS 前端工作量 | 中 | 优先方案①（复用 AG-UI 前端对接其 API），避免深改其 UI |

## 六、结论

**采用 DeerFlow 2.0 作为正式版底座是合理决策**：它的 MCP+Skills+子代理+多入口抽象与我们的目标形态一致，
底座同为 LangGraph，演示版两个月沉淀的工具层、安全机制、评测资产迁移成本接近零；
主要缺口（工具级审批、注册中心发现、GIS 前端、遥测播报）均有明确对策且工作量可估。
**先按 [03-POC计划.md](03-POC计划.md) 用 1~2 天验证三个硬指标，再启动正式开发。**
