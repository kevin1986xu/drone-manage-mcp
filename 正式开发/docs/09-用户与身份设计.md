# 用户与身份设计（docs/09）

> 2026-07-22 起草。补齐三关鉴权里唯一没落到「人」的一环：谁发起、谁批准、
> 以谁的身份读写平台。政务合规的核心诉求是**审计追责到人**与**越权有边界**。

## 1. 现状：有「租户」没有「人」

| 环节 | 现在的身份粒度 | 缺口 |
|---|---|---|
| 接入（关一） | 租户 key（`tenant-demo` = 一个组织） | 到不了「张三」这个人；租户内所有人共用一把 key |
| 操作（关二） | 确认单匿名（page_token 持有者=任何拿到链接的人） | 谁点的确认无从追溯；无法做发起人≠审批人 |
| 回源（关三） | 固定 admin（`DRONE_LOGIN_USERNAME=admin`） | Agent = 超管，越权无边界、审计无主体 |

**结论**：租户级隔离已就绪（Higress 消费者 key），但「人」这一层三处皆空。

## 2. 设计原则

1. **不新建用户库**。平台（若依 RuoYi）已有 `sys_user` / 角色 / `dataScope` /
   `sys_menu` 权限，政务客户账号都在里面。我们只做身份的**传递与消费**，不做管理。
2. **一条身份链，三个注入点**：用户身份从入口注入，全链路透传，在三关各自消费。
3. **Agent 权限 ≤ 用户权限**：回源带用户身份 → 平台 dataScope 自动过滤 →
   Agent 天然不可能越过发起人本身的权限（这是不越权的根本保证，不靠我们自觉）。
4. **两阶段推进**：先声明式（信任边界内，审计链先跑通），后可验证（OAuth 2.1，
   身份可验签）。不追求一步到位。

## 3. 身份链（三个注入点）

```
用户（若依账号）登录入口（BFF / webui / 对外网关）
   │  注入 X-Tenant-Key(组织) + X-User-Id(人)
   ▼
① 接入层  DeerFlow thread 绑定 user；ApiKeyMiddleware 已注入 scope.uav_tenant，
          扩展为 {tenant, user, scopes}；租户内角色 → 工具白名单
   │
   ▼
② 审批层  确认单记 initiated_by（发起对话的人）+ confirmed_by（点确认的人）；
          确认卡片页签名链接绑定 user；支持四眼原则（发起≠审批）
   │
   ▼
③ 回源层  set_platform_identity(user) → X-User-Id 透传平台；
          平台按该 user 的 dataScope 过滤 + sys_menu 校验高危工具权限
```

**现有资产盘点（大半已埋好）**：
- 关一：`auth.py` 的 `ApiKeyMiddleware` 已把租户注入 `scope.uav_tenant`，`UAV_TENANT_KEYS`
  的 meta 有 `scopes` 字段可扩展成角色→工具白名单；
- 关三：`drone_manage.py` 的 `set_platform_identity()` contextvar + `config.DRONE_USER_ID_HEADER`
  注入**已实现**（单测 `test_platform_auth.py` 覆盖），就差平台侧认这个头；
- 关二：审批服务确认单结构清晰，加两个字段即可。

## 4. 两阶段落地

### 阶段 1：声明式身份（信任边界内，本机可做，~1 天）

身份由入口「声明」——网关/BFF 说你是谁就是谁，安全性依赖服务间信任边界
（820x 收口 + 内网）。**目标：审计链先跑通，操作开始追责到人。**

| 改动点 | 内容 |
|---|---|
| 入口注入 | BFF / webui 登录态取 `userId` → 请求头 `X-User-Id`；对外网关（Higress）由消费方带 |
| 关一透传 | `ApiKeyMiddleware`：`scope.uav_tenant` 扩为 `{tenant, user, scopes}`；user 从 `X-User-Id` 取 |
| 关二审计 | 确认单加 `initiated_by` / `confirmed_by`；确认卡片页 page_token 签发时绑 user；审批日志记两者 |
| 关二四眼（可选开关） | 高危工具可配「发起人不得自批」——`confirmed_by != initiated_by` 才放行 |
| 关三透传 | 工具执行前 `set_platform_identity(scope 里的 user)`；`DRONE_USER_ID_HEADER=X-User-Id` 配上 |
| 审计串联 | Higress access log 加 `x_user`（已有 consumer 字段，再加一个）；mcp 侧 audit 拦截器记 user；三处 user 一致可端到端追溯 |

局限：身份是「声明」的，拿到内网+key 即可伪造 user——所以阶段 1 **必须配合 820x 收口**
（docs/07 信任边界②），只在网关背后可信。对外必须走阶段 2。

### 阶段 2：可验证身份（OAuth 2.1，对外产品形态）

按 docs/07 §4.1 阶段2 路线，身份可验签、不可伪造：

- 授权服务器：复用**若依用户体系**（政务客户已有账号，首选）或前置 Keycloak/Authing；
- 用户登录换 **JWT**：`sub`=用户、`aud` 绑定 MCP server（RFC 8707 防重放）；
- 全链路验签：Higress（JWT 插件，已内置）→ mcp-services 读 `sub` 做 user 身份 →
  回源换/透传平台 token；
- 工具可见性按 JWT scope 收敛（关一的角色白名单从「声明」变「验签」）。

到这一步，**标准 MCP 客户端（Claude 桌面端/Cursor）走 Authorization Code + PKCE
直连**——与 docs/08 通用前端合拢：通用客户端 + 可验证身份 = 真正对外产品形态。

## 5. 关键取舍

1. **紧急白名单⚡不限权**：返航/急停是止损动作，任何认证用户可触发（宁可误触发
   不可挡），但**审计必须记到人**（`emergency` 标记 + user）。
2. **四眼原则**默认关、可按工具配：演示/单人运维场景发起=审批同人，政务高危
   场景（如删围栏、改限高）可要求发起≠审批。
3. **租户 vs 用户职责分离**：租户 key 管组织隔离/限流/计费（Higress 侧，不下沉）；
   用户身份管审计/dataScope/权限（应用侧）。两者正交，不合并。
4. **dataScope 依赖平台侧**：关三真正生效需平台认 `X-User-Id` 并接 dataScope——
   这是唯一的外部依赖，阶段 1 我方先把头传到位、审计先记，平台就绪即闭环。
